**版本**: v1.0

### W1-A 仪表盘基础（3 天）

**目标**：用户无需导出文件，在 Web UI 内直接以图表形式查看 ADS 层数据。

**支持图表类型**：折线图、柱状图、数据表格、KPI 数字卡片

**图表配置参数**：
- `chart_type`：line / bar / table / kpi
- `x_field`：X 轴字段（时间轴或分类轴）
- `y_field`：Y 轴字段（数值字段）
- `agg_func`：SUM / AVG / COUNT / MAX / MIN（KPI 卡片用）
- `filter_expr`：附加过滤条件（DuckDB SQL WHERE 子句，可选）
- `title`：图表标题

**新增数据模型**：
```sql
CREATE TABLE etl_meta.dashboards (
  id            INT PRIMARY KEY AUTO_INCREMENT,
  name          VARCHAR(200) NOT NULL,
  description   TEXT,
  created_by    INT NOT NULL,
  created_at    DATETIME DEFAULT NOW()
);

CREATE TABLE etl_meta.chart_configs (
  id             INT PRIMARY KEY AUTO_INCREMENT,
  dashboard_id   INT NOT NULL,
  data_source_id INT NOT NULL,
  chart_type     ENUM('line','bar','table','kpi') NOT NULL,
  config_json    JSON NOT NULL,
  sort_order     INT DEFAULT 0,
  created_at     DATETIME DEFAULT NOW()
);
```

**新增 API**：

| 方法 | 路径 | 功能 |
|------|------|------|
| GET/POST | `/api/v1/dashboards` | 列出 / 创建仪表盘 |
| GET/PUT/DELETE | `/api/v1/dashboards/{id}` | 查询 / 更新 / 删除仪表盘 |
| POST | `/api/v1/dashboards/{id}/charts` | 添加图表 |
| PUT/DELETE | `/api/v1/dashboards/{id}/charts/{chart_id}` | 更新 / 删除图表 |
| GET | `/api/v1/dashboards/{id}/charts/{chart_id}/data` | 查询图表数据（DuckDB 执行）|

图表数据查询在后端由 DuckDB 完成，返回 `{columns, rows}` 格式，与现有 preview 端点一致。

**前端**：新增「仪表盘」页面（`/dashboard`）；图表渲染使用 `echarts-for-react`（Apache ECharts，MIT 协议）；图表配置使用侧边 Drawer 表单。

### W1-B 配置血缘（2 天）

**目标**：可视化展示每个数据源的完整字段变换链（Raw src_field → DWD dst_field → DWS 聚合列 → ADS 输出列），并记录每次映射配置变更历史。

**回答的问题**：「`amount` 字段在 DWD 中叫什么？经过什么聚合进入 DWS？ADS 最终输出的是哪个字段？」

**实现**：
- `mappings.py` PUT 端点在保存映射后，额外写入一条 `mapping_versions` 快照
- 血缘查询 API 读取当前 FieldMapping + AggRule + AdsRule，组装字段变换链返回前端
- 前端用 React Flow 渲染纵向字段列表，横向连线展示 4 层变换路径

**新增数据模型**：
```sql
CREATE TABLE etl_meta.mapping_versions (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  data_source_id  INT NOT NULL,
  snapshot_json   JSON NOT NULL,   -- 完整的 field_mappings 配置快照
  saved_by        INT NOT NULL,
  saved_at        DATETIME DEFAULT NOW()
);
```

**新增 API**：

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/v1/datasources/{ds_id}/lineage/config` | 返回当前字段变换链（raw→dwd→dws→ads）|
| GET | `/api/v1/datasources/{ds_id}/lineage/config/history` | 返回历史映射版本列表 |
| GET | `/api/v1/datasources/{ds_id}/lineage/config/diff?v1=&v2=` | 对比两个版本的字段映射差异 |

**前端**：新增「数据血缘」页面（`/lineage`）；使用 `@xyflow/react` 渲染字段变换链 DAG。

### W2-A 执行血缘（3 天）

**目标**：追溯某次 ADS 数据来自哪些原始上传文件，中间经过几次执行。

**回答的问题**：「本月报告的数据，来自 1 月 3 日上传的哪个 CSV 文件？」

**实现**：
- `etl_service.py` 执行 Raw→DWD 后，查询写入 DWD 的行对应的 `_run_id` / `_src_file`，写入 `execution_lineage`
- 血缘查询 API 组装 DAG 数据（源文件节点 → Raw → DWD → DWS → ADS）
- 前端在「数据血缘」页新增「执行血缘」Tab，节点显示执行时间 / 行数 / 源文件名

**新增数据模型**：
```sql
CREATE TABLE etl_meta.execution_lineage (
  id              INT PRIMARY KEY AUTO_INCREMENT,
  execution_id    INT NOT NULL,
  source_run_ids  JSON NOT NULL,   -- Raw 层消费的 _run_id 列表
  source_files    JSON NOT NULL,   -- 对应的 _src_file 列表
  recorded_at     DATETIME DEFAULT NOW()
);
```

**新增 API**：

| 方法 | 路径 | 功能 |
|------|------|------|
| GET | `/api/v1/datasources/{ds_id}/lineage/runs` | 返回执行血缘 DAG 数据 |
| GET | `/api/v1/datasources/{ds_id}/lineage/files` | 列出所有贡献过数据的源文件 |

## Wave 3 — 调度自动化（需新增 Prefect 服务）约 3 天

### W3-A Prefect 调度引擎 + SFTP 自动拉取（3 天）

**目标**：ETL 链路按 Cron 表达式自动执行，SFTP 无需手动浏览和触发。

**核心设计**：
- 每个 DataSource 可配置一条调度规则（Cron 表达式 + 最大重试次数）
- Prefect Flow = 单个 DataSource 的完整链路（SFTP/Upload → Raw → DWD → DWS → ADS）
- SFTP 自动拉取：按 `sftp_file_pattern` 过滤 → 与上次拉取记录对比 → 只拉取新文件，避免重复入库

**新增数据模型**：
```sql
CREATE TABLE etl_meta.schedules (
  id             INT PRIMARY KEY AUTO_INCREMENT,
  data_source_id INT NOT NULL,
  cron_expr      VARCHAR(100) NOT NULL,
  enabled        BOOLEAN DEFAULT TRUE,
  max_retries    INT DEFAULT 1,
  created_at     DATETIME DEFAULT NOW()
);

CREATE TABLE etl_meta.sftp_pull_log (
  id             INT PRIMARY KEY AUTO_INCREMENT,
  data_source_id INT NOT NULL,
  filename       VARCHAR(500) NOT NULL,
  pulled_at      DATETIME NOT NULL,
  run_id         VARCHAR(36) NOT NULL,
  rows_ingested  INT DEFAULT 0
);
```

**新增 Docker 服务**：`prefect-server`、`prefect-worker`（加入现有 compose）

**W3 完成后可追加**：仪表盘定时邮件推送（Prefect 定时任务 → 截图/数据 → 邮件）

## Wave 4 — 多租户隔离（可独立于 Wave 1-3 开发）约 5 天

---

### W4-A 数据质量报告（2 天）

**目标**：Raw→DWD 执行后自动生成数据质量报告，暴露空值率、类型转换失败率等问题。

**报告维度**：空值率 / 类型转换失败率 / 唯一值数 / 数值列分布（min/max/avg）/ Schema 变更检测

**存储**：新增 `etl_meta.dq_reports` 表；在执行历史详情页展示报告。

---

### W4-B 工作空间隔离 + 角色权限（3 天）

**目标**：多团队各自管理独立的数据源和 ETL 配置，互不可见。

**设计**：
- 新增 `etl_meta.workspaces` 表
- 所有现有表增加 `workspace_id` 外键（现有数据归入 workspace_id=1，零停机迁移）
- 用户角色：`admin`（全局）/ `workspace_admin`（空间管理员）/ `viewer`（只读）
- JWT 中携带 `workspace_id`；所有 API 查询自动过滤当前空间

**W4-B 完成后**：仪表盘自动获得多租户隔离支持（补充 dashboard.workspace_id 字段）

---

## Wave 5 — 体验提升（优先级最低）约 4 天

---

### W5-A React Flow ETL 拖拽画布（4 天）

**目标**：用拖拽式 DAG 画布替代当前表单式配置页，让数据流关系一目了然。

**实现策略**：Phase 1 所有配置保持原有数据模型不变；画布和表单 UI 并存，用户可选择使用哪种。
