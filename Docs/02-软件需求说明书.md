## 1 引言

### 1.1 目的

本文档为 DataETL2 平台的设计、开发、测试和验收提供完整的功能与非功能需求基准。文档读者包括产品负责人、前后端开发工程师、测试工程师及运维团队。

### 1.2 项目范围

**产品名称**: DataETL2

**核心能力**:
- 可视化数据管道构建（拖拽式，低代码）
- 多数据源连接与数据提取
- 规则化数据转换与清洗
- 灵活的数据目标写入
- 定时调度与执行引擎
- 监控、告警与审计

**明确排除（不在本版本范围内）**:
- 数据仓库/数据湖底层存储
- BI 报表与数据可视化
- 机器学习特征工程
- 实时流式处理（初期仅批处理）

### 1.3 定义与缩略词

| 术语 | 定义 |
|------|------|
| ETL | Extract, Transform, Load — 数据抽取、转换、加载的统称 |
| 数据源 (Data Source) | 用户在系统中定义的数据接入配置，包含接入类型（SFTP/手动上传）、目标表名等，所有名称由用户自定义 |
| 字段映射 (Field Mapping) | 将 Raw 层字段名/类型转换为 DWD 层目标字段的规则配置，由用户在界面中配置 |
| 运行 (Run) | 某次执行的实例，拥有唯一 Run ID，关联来源文件和执行结果 |
| etl_raw | 原始数据层，文件入库后原样保存，永久保留，用于追溯和重跑 |
| etl_dwd | 明细数据层，经字段映射和过滤规则清洗后的标准数据 |
| etl_dws | 汇总数据层，对 DWD 数据按用户配置的聚合规则汇总 |
| etl_ads | 应用数据层，最终输出层，用于报表、导出或下游消费 |
| SFTP | SSH 文件传输协议，服务商将 CSV/Excel 文件放置在 SFTP 服务器，系统自动或手动拉取 |
| 零硬编码原则 | 系统不预设任何业务表名、字段名、数据源名称，所有业务相关名称由用户在界面中配置 |
| 工作空间 (Workspace) | 用户/团队的独立资源隔离空间（Phase 2 功能）|
| RBAC | 基于角色的访问控制（Role-Based Access Control） |
| SLA | 服务级别协议（Service Level Agreement） |
| SSO | 单点登录（Single Sign-On） |
| MVP | 最小可行产品（Minimum Viable Product） |

## 2 总体描述

### 2.1 产品视角

DataETL2 是一个独立部署的 Web 应用系统。其系统上下文边界如下图所示：

```
┌──────────────────────────────────────────────────────────────┐
│                      DataETL2 系统边界（Phase 1）              │
│                                                              │
│  ┌──────────────┐       ┌──────────────┐                     │
│  │  Web 前端     │◄─────►│  FastAPI     │                     │
│  │  (React 18)  │       │  REST API    │                     │
│  └──────────────┘       └──────┬───────┘                     │
│                                │                            │
│                     ┌──────────┼──────────┐                  │
│                     ▼          ▼          ▼                  │
│               ┌──────────┐ ┌──────┐ ┌─────────────────────┐ │
│               │  DuckDB  │ │MySQL │ │  paramiko (SFTP 客户端)│ │
│               │ 内存处理  │ │唯一  │ │                      │ │
│               │ 引擎     │ │持久化│ └──────────────────────┘ │
│               └──────────┘ │存储  │                          │
│                    │       │      │                          │
│                    └──────►│etl_meta / etl_raw              │ │
│                            │etl_dwd / etl_dws / etl_ads     │ │
│                            └──────────────────────┘         │
└──────────────────────────────────────────────────────────────┘

外部系统：
  - SFTP 服务器（服务商放置 CSV/Excel 文件）
  - SMTP 邮件服务器（执行失败告警）
```

**数仓分层**：所有表名由用户在系统中自定义，系统不预设任何业务表名。

```
文件（CSV/Excel）
    │ SFTP 拉取 / 手动上传
    ▼
etl_raw.<用户定义表名>    原始数据，永久保留（含 _src_file/_ingested_at/_run_id）
    │ 字段映射 + 过滤规则
    ▼
etl_dwd.<用户定义表名>    清洗后明细数据
    │ 聚合规则（GROUP BY + 聚合函数）
    ▼
etl_dws.<用户定义表名>    汇总数据
    │ 输出规则（字段选取 + 排序）
    ▼
etl_ads.<用户定义表名>    应用层（可导出 CSV/Excel）
```

### 2.2 产品功能概述

DataETL2 提供以下八大功能分组（详见 `01-功能列表.md`）：

| 序号 | 功能分组 | 核心能力 |
|------|---------|---------|
| 1 | 数据源连接管理 | 配置、测试、管理各类数据源连接 |
| 2 | 可视化管道设计 | 拖拽式构建 ETL 数据流图 |
| 3 | 数据转换处理 | 丰富的无代码/低代码转换节点库 |
| 4 | 数据目标写入 | 向多种目标系统写出处理结果 |
| 5 | 调度与执行引擎 | 定时、依赖、手动触发，并发/重试控制 |
| 6 | 监控与告警 | 执行历史、日志、仪表盘、邮件/Webhook 告警 |
| 7 | 用户权限管理 | RBAC、工作空间隔离、API Token |
| 8 | 系统管理 | 系统配置、健康检查、存储管理 |

### 2.3 用户类别与特征

| 用户类别 | 技术水平 | 主要使用场景 | 关键关注点 |
|---------|---------|------------|-----------|
| **业务分析师**（主要用户） | 低-中等，无编程能力 | 构建和运行管道，查看执行结果 | 易用性、无代码操作 |
| **数据工程师**（高级用户） | 高，熟悉 SQL/Python | 构建复杂管道，API 集成，自定义转换 | 灵活性、可扩展性 |
| **系统管理员** | 高，具备运维能力 | 系统配置、用户管理、监控平台健康 | 稳定性、安全性 |
| **管理者/决策者** | 低 | 查看执行状态汇总和统计数据 | 数据可见性、汇总报告 |

### 2.4 运行环境

**客户端**:
- 支持浏览器：Chrome 100+、Edge 100+、Firefox 100+、Safari 15+
- PC 端优先，移动端支持查看功能（只读）

**服务端**:
- 操作系统：Linux（Ubuntu 22.04 LTS / CentOS 8+）
- 容器化：支持 Docker Compose（开发/小规模）和 Kubernetes（生产）
- 最低配置：4 Core CPU、8 GB RAM、100 GB SSD
- 推荐生产配置：8 Core CPU、32 GB RAM、1 TB SSD

### 2.5 设计与实现约束

| 约束 ID | 约束内容 |
|---------|---------|
| C-01 | 前端使用 React 18+，后端使用 Python 3.11+ / FastAPI |
| C-02 | Phase 1 仅支持手动触发执行；定时调度（Prefect 3.x）在 Phase 2 引入 |
| C-03 | 所有数据（元数据 + 四层数仓数据）持久化在 MySQL 8.0（Docker 容器内新建） |
| C-04 | 数据处理使用 DuckDB 内嵌引擎（内存处理，不持久化），用户不感知 |
| C-05 | 所有 API 遵循 RESTful 设计规范，返回 JSON |
| C-06 | 前后端分离架构，通过 HTTP API 通信（开发阶段），生产环境通过 Nginx 统一入口 |
| C-07 | Phase 1 部署方式：Docker Compose（4 容器：nginx / frontend / api / mysql）|
| C-08 | 所有敏感凭据（SFTP 密码等）使用 Fernet 对称加密存储，API 响应不返回明文 |
| C-09 | 零硬编码原则：系统不预设任何业务表名、字段名、数据源名称；全部由用户配置 |
| C-10 | SFTP 连接使用 paramiko 库；可视化画布（React Flow）推迟至 Phase 2 |

### 2.6 假设与依赖

**假设**:

| 假设 ID | 内容 |
|---------|------|
| ASS-01 | 用户具备基本数据概念（行、列、表），无需编程能力 |
| ASS-02 | 目标数据库等系统由用户自行管理，平台只负责连接和读写 |
| ASS-03 | 初期面向中小规模数据，单次运行数据量不超过 1000 万行 |
| ASS-04 | 部署环境网络稳定，平台与数据源之间延迟可接受（< 200ms） |

**依赖**:

| 依赖 ID | 内容 |
|---------|------|
| DEP-01 | 邮件告警依赖外部 SMTP 服务 |
| DEP-02 | 企业身份认证可选依赖 LDAP/SAML SSO 服务 |
| DEP-03 | 云存储连接依赖用户提供有效的云服务凭据 |

## 3 具体需求

### 3.1 外部接口需求

#### 3.1.1 用户界面需求

| ID | 需求内容 |
|----|---------|
| UI-01 | 主界面布局：左侧导航 + 顶部操作栏 + 中央内容区，标准 Web 应用布局 |
| UI-02 | 管道设计器采用全屏画布模式，支持鼠标滚轮缩放（10%~200%）、拖拽移动、节点连线 |
| UI-03 | 操作响应时间提示：加载超过 500ms 显示进度指示器；超过 3s 显示取消选项 |
| UI-04 | 支持中文界面，预留国际化扩展能力（i18n） |
| UI-05 | 所有破坏性操作（删除管道、停止执行）需弹窗二次确认 |
| UI-06 | 错误提示信息对用户友好，不显示技术堆栈信息（如 Python traceback） |

#### 3.1.2 REST API 接口需求

| ID | 需求内容 |
|----|---------|
| API-01 | 提供完整 REST API，覆盖所有平台核心功能 |
| API-02 | 使用 Bearer Token（JWT）进行 API 认证 |
| API-03 | API 版本化，路径前缀为 `/api/v1/` |
| API-04 | 提供 OpenAPI 3.0 规范文档，可通过 `/docs` 访问 Swagger UI |
| API-05 | 统一响应格式：`{"data": ..., "error": null, "meta": {"page": ..., "total": ...}}` |
| API-06 | 实施速率限制（Rate Limiting），默认每用户 1000 次请求/分钟 |

#### 3.1.3 外部系统接口

| ID | 需求内容 |
|----|---------|
| EXT-01 | 数据库连接支持主流数据库原生协议（PostgreSQL protocol、MySQL protocol 等） |
| EXT-02 | 云存储使用官方 SDK（boto3、azure-storage-blob、google-cloud-storage） |
| EXT-03 | Webhook 通知使用标准 HTTP POST，请求体为 JSON，支持自定义 Headers |
| EXT-04 | 邮件通知通过 SMTP 协议发送，支持 TLS 加密 |

### 3.2 功能需求

功能需求条目格式：`FR-[模块]-[序号]: 功能名称`

#### 模块 A — 数据源连接管理

**FR-A-01: 数据源注册向导**
- 描述：提供分步引导界面，允许用户配置数据源连接参数
- 输入：数据源类型选择、连接参数（主机/端口/用户名/密码/数据库名等）
- 处理：验证参数完整性，尝试建立测试连接
- 输出：成功则保存并提示；失败则显示具体错误（超时/认证失败/网络不可达）
- 优先级：M | 关联用例：UC-21

**FR-A-02: 关系型数据库连接**
- 描述：支持通过标准协议连接 MySQL、PostgreSQL、SQL Server、Oracle、SQLite
- 输入：各数据库类型对应的连接参数
- 处理：使用对应驱动建立连接，执行 `SELECT 1` 验证
- 输出：连接成功确认，含数据库版本信息
- 优先级：M | 关联用例：UC-21、UC-22

**FR-A-03: 文件源连接**
- 描述：支持用户上传 CSV、Excel (.xlsx/.xls)、JSON、Parquet 文件作为数据源
- 输入：文件上传（≤ 200MB），可选编码/分隔符/Sheet 配置
- 处理：解析文件，推断字段类型，生成预览
- 输出：文件解析成功，显示列名和前 5 行预览
- 优先级：M | 关联用例：UC-21

**FR-A-04: REST API 连接**
- 描述：通过 URL + 认证配置拉取 HTTP/HTTPS API 数据
- 输入：URL、HTTP 方法、请求头、认证方式（None/API Key/Bearer Token/Basic Auth）、分页配置
- 处理：发起请求，解析 JSON/XML 响应，映射到表格结构
- 输出：解析后的数据预览
- 优先级：M | 关联用例：UC-21

**FR-A-11: 连接测试与诊断**
- 描述：独立的连接测试功能，提供详细诊断信息
- 输入：已填写的连接参数
- 处理：尝试连接，记录各阶段（DNS 解析/TCP 连接/认证/查询）耗时
- 输出：成功时显示连接耗时；失败时显示失败阶段和具体错误码
- 优先级：M | 关联用例：UC-22

**FR-A-12: 凭据加密存储**
- 描述：所有密码、API Key、私钥等敏感凭据加密后存储，不以明文出现在 DB、日志或 API 响应中
- 实现要求：AES-256-GCM 加密，密钥通过环境变量注入
- 优先级：M

#### 模块 B — 管道设计器

**FR-B-01: 可视化拖拽画布**
- 描述：基于 DAG 的可视化管道编辑器，支持节点拖拽、连线和移动
- 输入：用户鼠标操作（拖拽节点、拖动连线端点）
- 处理：维护管道 DAG 结构，实时验证连线合法性（防止循环）
- 输出：更新画布显示，DAG 结构同步到服务端草稿
- 优先级：M | 关联用例：UC-32

**FR-B-03: 节点配置侧面板**
- 描述：点击任意节点，右侧滑出配置表单，根据节点类型动态渲染字段
- 输入：节点点击事件，用户填写配置表单
- 处理：验证配置有效性，保存到节点配置
- 输出：节点标题显示配置摘要（如源节点显示数据源名称）
- 优先级：M | 关联用例：UC-33

**FR-B-04: 管道保存/发布**
- 描述：草稿状态随时保存，发布操作验证完整性后生效
- 输入：用户点击「保存」或「发布」
- 处理：保存验证管道结构（所有节点已连接，无孤立节点，有且仅有一个起点）；发布创建版本号
- 输出：草稿持久化 / 发布成功并显示版本号
- 优先级：M | 关联用例：UC-35

**FR-B-12: 实时数据预览**
- 描述：在设计时可在任意节点触发预览，查看该节点的输出数据样本
- 输入：用户点击节点的「预览」按钮
- 处理：执行从起点到该节点的所有操作，抽取样本数据（最多 100 行）
- 输出：在面板展示数据表格，含行数、列名、数据类型
- 优先级：S | 关联用例：UC-34

#### 模块 C — 数据转换节点

**FR-C-01: 字段映射**
- 描述：源字段到目标字段的名称重命名和数据类型转换配置
- 输入：源列名列表（自动推断）、目标列名、目标类型
- 处理：构建映射表，转换时应用重命名和类型转换
- 优先级：M | 关联用例：UC-33

**FR-C-03: 行过滤**
- 描述：基于条件表达式过滤数据行
- 输入：条件构建器（字段 + 运算符 + 值），支持 AND/OR 逻辑组合
- 处理：生成过滤谓词，应用到数据集
- 输出：符合条件的行；过滤掉的行数统计
- 优先级：M | 关联用例：UC-33

**FR-C-06: 聚合/分组**
- 描述：按一个或多个字段分组，对指定字段执行聚合运算
- 输入：GROUP BY 字段列表、聚合函数配置（字段 + 函数 + 输出别名）
- 处理：执行 GROUP BY 聚合运算（SUM/AVG/COUNT/MAX/MIN/COUNT DISTINCT）
- 优先级：M | 关联用例：UC-33

**FR-C-07: 数据关联 (Join)**
- 描述：将两个节点的数据集按指定键关联合并
- 输入：左表节点、右表节点、Join 类型（INNER/LEFT/RIGHT/FULL）、关联键字段对
- 处理：执行 Join 操作，合并结果列集
- 优先级：M | 关联用例：UC-33

**FR-C-15: 自定义 SQL 转换**
- 描述：允许用户在节点输入 SQL 语句对数据进行转换
- 输入：SQL 语句（使用 `{{input}}` 引用上游数据）
- 处理：在 DuckDB 中对上游数据创建临时视图，执行 SQL
- 输出：SQL 查询结果
- 约束：禁止 DDL（CREATE TABLE/DROP 等），禁止访问本地文件系统
- 优先级：S | 关联用例：UC-33

**FR-C-17: 数据验证节点**
- 描述：对数据执行规则校验，不合格行路由到单独的错误输出端口
- 输入：验证规则配置（非空/唯一/范围/正则/自定义表达式）
- 处理：逐行验证，合格行流入主输出，不合格行（含错误原因字段）流入错误输出
- 输出：合格数据 + 错误数据（可连接告警或写入错误日志目标）
- 优先级：S | 关联用例：UC-33

#### 模块 E — 调度与执行

**FR-E-01: 手动触发执行**
- 描述：用户可随时手动触发管道立即执行
- 输入：用户点击「立即执行」，可选传入覆盖参数
- 处理：校验管道状态（必须为已发布），提交执行任务
- 输出：返回 Run ID，界面跳转至执行状态页
- 优先级：M | 关联用例：UC-41

**FR-E-02: Cron 表达式调度**
- 描述：配置 Cron 表达式定时触发管道执行
- 输入：Cron 表达式（含可视化 Cron 编辑器辅助填写），时区设置
- 处理：注册到调度引擎，计算并展示下次执行时间
- 输出：调度配置保存，显示下 5 次执行时间预览
- 优先级：M | 关联用例：UC-42

**FR-E-09: 重试策略**
- 描述：配置管道执行失败后的自动重试行为
- 输入：最大重试次数（0-10）、重试间隔（秒）、是否指数退避
- 处理：执行失败时按策略自动重试，重试次数记录在执行日志中
- 优先级：S | 关联用例：UC-46

#### 模块 F — 监控与告警

**FR-F-01: 执行历史列表**
- 描述：列出所有管道的历史执行记录，支持按状态/时间/管道名称过滤
- 输入：过滤条件、分页参数
- 输出：执行记录列表（管道名/触发时间/耗时/状态/处理行数）
- 优先级：M | 关联用例：UC-51

**FR-F-02: 执行详情与日志**
- 描述：查看单次执行的完整信息，包括各节点耗时、日志输出、错误信息
- 输入：Run ID
- 输出：节点级别的执行时间线、结构化日志（时间戳/级别/消息）
- 优先级：M | 关联用例：UC-52

**FR-F-05: 失败告警 — 邮件**
- 描述：管道执行失败时，自动向预配置收件人发送告警邮件
- 输入：失败事件、告警接收人列表（在管道配置中设置）
- 处理：渲染告警邮件模板（包含管道名、失败时间、错误摘要、日志链接）
- 输出：邮件发送成功/失败状态记录在系统日志中
- 优先级：M | 关联用例：UC-54

#### 模块 G — 用户权限管理

**FR-G-01: 用户注册/登录**
- 描述：支持账号密码登录，可选 SSO/LDAP 集成
- 输入：用户名/密码 或 SSO Token
- 处理：验证凭据，生成 JWT（有效期 8 小时），支持 Refresh Token（7 天）
- 输出：登录成功返回 Token；失败返回错误原因（不透露是账号不存在还是密码错误）
- 优先级：M | 关联用例：UC-11

**FR-G-02: 角色权限控制**
- 描述：预定义四种系统角色，控制功能访问权限
- 角色定义：

| 角色 | 描述 | 可用功能 |
|------|------|---------|
| 管理员 (Admin) | 最高权限 | 全部功能 + 用户管理 + 系统配置 |
| 编辑者 (Editor) | 创建和编辑 | 创建/编辑/删除管道和数据源、触发执行 |
| 执行者 (Operator) | 只能运行 | 查看管道、触发执行、查看日志 |
| 查看者 (Viewer) | 只读 | 查看管道配置、执行历史（不可触发） |

- 优先级：M | 关联用例：UC-62

         React 前端              FastAPI 后端

                    读 CSV/Excel               ├── etl_meta

                    结果写入 MySQL →           ├── etl_dwd

### 设计原则

1. **零硬编码**：系统不预设任何业务表名、字段名、数据源名称，全部由用户配置。
2. **Raw 永久保留**：etl_raw 层数据不自动删除，支持随时重跑。
3. **DuckDB 透明化**：DuckDB 只是内存处理工具，所有数据最终存 MySQL，用户无感知。
4. **凭据安全**：SFTP 密码 Fernet 加密存储，API 响应不返回明文密码。
